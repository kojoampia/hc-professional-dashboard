# Phase 4 Backend Contract Reconciliation

## Scope and boundary

The models in `src/main/webapp/app/health-connect/health-connect.models.ts` are frontend-only contracts. `MockHealthConnectRepository` is an in-memory adapter selected through `HEALTH_CONNECT_REPOSITORY`; it does not send HTTP requests or claim backend authorization. A future HTTP adapter must use `ApplicationConfigService.getEndpointFor(...)` and replace that token binding only after the listed contracts are approved.

Status meanings:

- **Existing**: a generated professionalMS client and endpoint path are present in this frontend.
- **Missing**: no generated client/model expresses the required feature contract.
- **Awaiting confirmation**: an adjacent generated shape exists, but its relationship, semantics, or API fields cannot safely be assumed.

## Existing endpoint inventory

The generated professionalMS service classes currently construct these endpoint paths through `ApplicationConfigService`: `api/profiles`, `api/addresses`, `api/documents`, `api/reports`, `api/tasks`, `api/medications`, `api/teams`, `api/memberships`, `api/metadata`, `api/stats`, `api/hc-credentials`, and `api/hc-pay-options`. Their presence is client code only; this document does not assert that a deployed backend supports every route or the feature relationships above.

## Integration sequence

1. Confirm patient/profile ownership and contact relation, then add an HTTP patient adapter.
2. Approve distinct clinical case, activity, report, recommendation, and duty-roster APIs.
3. Confirm custom authority provisioning and backend mutation authorization.
4. Replace `HEALTH_CONNECT_REPOSITORY` with an HTTP implementation while retaining the models, selectors, and local test fixture adapter.
